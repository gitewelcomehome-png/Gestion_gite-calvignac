Scalabilité

Retour Surveillance

Cible

Accompagner la montée en charge jusqu’à 70 000 utilisateurs avec des critères GO/NO-GO clairs à chaque palier.

Paliers opérationnels

Palier utilisateurs actifs Objectif principal Actions à faire Déclencheur de passage
0 → 500 Fiabilité de base Monitoring erreurs transport, check hebdo API-only, KPI DB hebdo. 4 semaines stables (<1% erreurs, p95 < 500ms)
500 → 2 000 Stabiliser pics standards Rate-limit sur flux critiques, cache lectures fréquentes, retry idempotent uniquement. 2 runs de charge PASS consécutifs
2 000 → 10 000 Tenue API + DB Audit index/FK, optimisation requêtes lentes, suivi dead tuples renforcé. Worst p95 < 350ms, 0 query > 30s
10 000 → 25 000 Robustesse sous charge continue Soak test 4–8h, alerting proactif, runbook incident prêt et testé. Soak PASS sans dérive ni alerte critique
25 000 → 50 000 Pré-scaling Réduction coûts route chaude, durcissement quotas/limites, tests rollback + reprise. 3 cycles release sans régression + 1 stress test PASS
50 000 → 70 000 Industrialisation Tests de charge planifiés, budget perf par route, gouvernance perf hebdo, plan capacité documenté. 2 campagnes PASS successives à cible haute + SLO respectés

Checklist minimale par palier

  • Run API-only à jour avec verdict PASS.
  • Script SQL hebdo exécuté (dead tuples, index non utilisés, requêtes longues).
  • Plan de rollback vérifié sur la release en cours.
  • Alerte erreurs transport active (seuil: 1%).
